# Spec Synchronization Gaps

As of 2026-06-27, local docs are ~90% aligned with spec but have specific gaps:

## Critical (Spec Compliance)

1. **Thread ID vs Group ID Confusion** (`architecture.md:248`)
   - **Current:** "Thread ID = SHA256('group:' + thread_id)" — circular definition
   - **Should be:** "Thread ID = ULID (generated by sender); MLS Group ID = SHA-256('group:' || thread_id)"
   - **Spec:** core.md references this clearly, encryption.md §Groups and Threads defines derivation

2. **Server Signature Binding String** (not documented)
   - **Missing:** Exact format of `UCPHelloAck.server_sig` binding string
   - **Spec says:** `"server_hello:" || auth_token || server_id` (core.md §Negotiation)
   - **Impact:** Implementation detail needed for server connection verification

3. **API Spec Not Implemented** (README mentions `spec/api.md`)
   - **Status:** Listed in README as deliverable but does not exist in https://github.com/unifiedcommunicationsprotocol/spec
   - **Impact:** No reference for REST endpoints, wire encoding, push events
   - **Workaround:** Use `IMPLEMENTATION.md` and `api.ts` stubs as guide; defer formal spec

## Important (Completeness)

4. **Recipient Preferences Caching**
   - **Current:** constraints.md mentions preferences but not caching strategy
   - **Spec:** preferences.md §Updating Preferences — clients should cache for session and re-fetch at session start
   - **Action:** Document in IMPLEMENTATION.md that `preferences` are fetched once and cached

5. **Stale Key Share Handling Details**
   - **Current:** Mentioned in constraints, not detailed
   - **Spec:** ai.md §Stale Key Share Handling — detailed client/server protocol for epoch advance while offline
   - **Action:** Add to crypto implementation plan in docs/mls-implementation.md

6. **BCC Group ID Derivation** (correct but could clarify)
   - **Current:** architecture.md mentions separate BCC groups
   - **Spec:** encryption.md — `SHA-256("group_bcc:" || thread_id || ":" || bcc_recipient_address)`
   - **Action:** Clarify in BCC section that group ID includes recipient address

7. **Three-Phase Verification Format** (architecture.md §Data Flow: Receiving)
   - **Current:** Documented but lacks specific error handling
   - **Spec:** core.md §Identity Verification — Phase 1 before MLS, Phase 2 after MLS; rejection on any failure
   - **Action:** Add "reject message, discard plaintext" explicitly

8. **Content-Type Values** (application payload)
   - **Missing:** Spec defines 6 content types (message, receipt, edit, delete, attachment, plus reserved 0x02 for reaction in 1.1)
   - **Action:** Add to encryption.md or IMPLEMENTATION.md with note that 0x02 MUST be rejected as unknown in 1.0

9. **Attachment Epoch Integrity**
   - **Missing:** Server verification that attachment epoch matches message envelope epoch
   - **Spec:** messages.md §Attachments — server rejects with `attachment_epoch_mismatch` if mismatch detected
   - **Action:** Add to api.ts stubs or IMPLEMENTATION.md

10. **Forward Compatibility Rules**
    - **Current:** Mentioned generically
    - **Spec:** messages.md §Forward Compatibility — unknown fields at any nesting level MUST be preserved, not stripped
    - **Action:** Document JSON serialization rules (sort keys, preserve unknowns, handle in all layers)

## Minor (Implementation Guidance)

11. **Bridge Attestation DKIM Field**
    - **Missing:** `dkim: "pass | fail | none"` optional field in bridge attestation
    - **Spec:** bridge.md §DKIM Verification
    - **Action:** Note in bridge implementation plan

12. **System Messages**
    - **Partially documented:** delivery_failure mentioned, member_added/member_removed not in docs
    - **Spec:** messages.md §System Messages — defines 3 types
    - **Action:** Add to message schema documentation

13. **UCPWelcomeExtension Type Value Warning**
    - **Missing:** Spec notes `0x0F01` is IANA placeholder; production blocker
    - **Spec:** encryption.md §UCPWelcomeExtension
    - **Action:** Add build flag/warning to prevent shipping unregistered value

14. **Read Receipts Never for Bridge Messages**
    - **Missing:** Constraint that receipts must NOT be sent for bridge-attested messages
    - **Spec:** messages.md §Read Receipts
    - **Action:** Add to constraints or IMPLEMENTATION.md

15. **Bundle Idempotency Log Retention**
    - **Missing:** Server bundle log must retain entries 72 hours minimum after commit
    - **Spec:** core.md §Bundle Idempotency
    - **Action:** Note for server reference, clients just need to know they can retry indefinitely
